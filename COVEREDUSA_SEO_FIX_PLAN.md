# CoveredUSA SEO Pipeline Fix — Execution Plan

**Created:** 2026-05-15
**Author:** Frank (technical lead) for Jacob (CEO)
**Goal:** Move the daily article pipeline from "writer + fact-checker that holds 50% of articles" to "writer + editor that ships 100% of articles, every day, no human review required."

---

## Overview

Today's cron held 5 of 10 articles. Of the 5 that "shipped," 2 have unfixed style issues live on the site right now. The verifier was designed to be cautious — it can fix small fact drift but is forbidden from touching anything structural, and "held" articles wait for human review. You've said: no human review, ever; ship everything; the verifier should act as an editor.

This plan makes five changes:

1. **Verifier becomes an editor.** Broader auto-fix scope. Narrow carve-out for editing the slug + title + description fields of frontmatter (currently completely forbidden). No prose invention — when an article needs new content written, the verifier sends it back to the writer.
2. **"Held" status is eliminated.** Every article ships. When the verifier can't auto-fix something, the article goes back to the writer for one regeneration attempt. If that still fails, it ships with a flag noted in the report.
3. **Writer regen with 1-retry cap.** Stage 1 cron gets a new step between verify and sheet-update that re-spawns the writer with explicit feedback when needed.
4. **Opus 4.7 on writer + verifier.** Smarter models for the parts that reason about content. Orchestrator + Stage 2 stay on Sonnet (cheap plumbing).
5. **Backfill today's 5 shipped articles** with the new editor-mode verifier so the live em-dash/citation issues get fixed. Re-run today's 5 held articles through the new system so they ship too.

## TASK 1: Rewrite the Verifier (Major)

- **Status:** todo
- **Difficulty:** Hard
- **File:** `.claude/agents/coveredusa-article-verifier.md`
- **Estimated time:** 25 min

### What to Do

Rewrite the verifier prompt from "fact-checker with detect-only on structure" to "editor with broader auto-fix scope, sends back to writer when it can't fix something itself."

**Specific structural changes to the prompt:**

1. **Frontmatter changes:**
   - `model: sonnet` → `model: claude-opus-4-7` (Opus 4.7 with 1M context)
   - Update the description field to reflect editor-mode behavior
   - Keep `background: true`, `permissionMode: bypassPermissions`, `maxTurns: 50`

2. **Reframe STEP 0–STEP 2.5** intro language:
   - Remove "dual-purpose verifier" language
   - Replace with "editor mode: you make fixes, you don't write new sections. When an article needs new content written from scratch, return needs-regen and the orchestrator will re-spawn the writer."

3. **Expand "what the verifier MAY edit" (replace the implicit negative list with an explicit positive list):**

   **Body content (existing scope, unchanged):**
   - Numeric corrections against primary sources
   - Em-dash (`—`) removal
   - Double-hyphen (`--`) removal — narrow string replacement only
   - Banned filler phrase removal
   - Paragraph-opener pronoun fix (only when the named entity is in the same paragraph — narrow sentence-level rewrite)

   **Frontmatter (NEW narrow carve-out):**
   - `slug` field: strip year regex `\b(19|20)\d{2}\b`, collapse hyphens, trim. After the edit, `mv` the .md file to match the new slug. Check for filename collision first; if collision, ship with flag instead of renaming.
   - `title` field: replace `--` with comma/period/parens per local context
   - `description` field: replace `--` with comma/period/parens per local context
   - NO OTHER frontmatter fields may be edited. Date, keywords, target, topicCluster, keyTerms, isLighthouse, isDeprecated all remain off-limits.

   **Inline citation upgrade (NEW):**
   - When the article cites a source in plain text (e.g., "a 2024 Health Affairs study found..."), the verifier MAY do a WebSearch to find the actual URL and wrap the existing text in markdown link syntax: `[a 2024 Health Affairs study](https://...)`.
   - This is wrapping existing words, not writing new ones. Maximum 3 citation upgrades per article (don't go overboard).
   - If the verifier can't find a clean URL after 1 search attempt per citation, skip it.

   **Income-table row fill (NEW):**
   - When an article has an income-gated table with fewer than 9 rows (sizes 1-8 + each-additional), the verifier may fill in missing rows using WebFetch on ASPE poverty-guidelines page. Only for FPL-based tables where the existing rows are already correct. If existing rows are wrong, fix them first, then fill.

4. **Define the new "needs-regen" return status (replaces "held"):**

   The verifier returns `status: "needs-regen"` ONLY when:
   - A required prose section is missing entirely (How-to-Apply section, FAQ section, state-by-state table)
   - The article body is fundamentally off-topic from the keyword
   - The income-table is missing entirely (not just short-rowed)
   - 0 `.gov` citations exist AND no plain-text source mentions are present to upgrade

   The verifier returns `status: "approved" | "corrected" | "shipped-with-flag"` otherwise. The `"held"` status is removed from the schema entirely.

5. **Update the return JSON shape:**

   ```json
   {
     "slug": "the-slug",
     "status": "approved | corrected | shipped-with-flag | needs-regen | error",
     "claims_checked": N,
     "claims_corrected": N,
     "claims_flagged": N,
     "change_log": [...],
     "flags": [...],
     "regen_reason": "only present when status=needs-regen — explicit text the writer can act on"
   }
   ```

   The `gates` and `gates_failed` fields can stay as informational tracking, but the cron only branches on `status`.

6. **Remove all "HOLD" and "do NOT ship" language** throughout the prompt.

7. **Strengthen the GATE D auto-fix instructions** (the one that's currently broken on 100% of articles):
   - Add a hard rule: "Before returning JSON, run `grep -c -- '--' <file-body-only>`. If count > 0, you have NOT completed your task. Continue fixing until count = 0."
   - This is a post-condition assertion the model has to satisfy before exiting.

### Success Criteria

- [ ] New prompt does NOT contain the strings "held", "HOLD", or "do NOT ship" outside of historical-context notes
- [ ] New prompt contains explicit positive list of editable frontmatter fields (slug, title, description) with explicit constraints
- [ ] Return JSON schema documented with `needs-regen` status, not `held`
- [ ] GATE D auto-fix has a post-condition grep assertion
- [ ] Inline-citation-upgrade behavior documented with max-3-per-article cap
- [ ] Income-table row-fill behavior documented for FPL-only case
- [ ] Frontmatter: `model: claude-opus-4-7`
